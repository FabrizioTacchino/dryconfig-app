# F20.7 — Arrotondamento mismatch create vs bulk update

**Data risoluzione**: 2026-05-13
**Severità**: 🟡 media (delta di centesimi, visibile su totali grandi)
**Commit**: `0cf8708`

## Sintomo

Dopo F20.6, residuo delta di centesimi al "Aggiorna prezzi":
- ac1: preview €13.41/m² (no finish)
- ac2: estimate totale €184.15
- ac3: dopo aggiorna prezzi €184.10
- Delta: **€0.05 esatti** su 10 m²

## Root cause

Tre pipeline con arrotondamento diverso:

| Pipeline | unit_cost | total_cost |
|---|---|---|
| Preview (dialog) | full-precision (es. 18.41495) | full-precision (184.1495) |
| Create (insert) | full-precision (18.41495) | area × full-precision = 184.1495 |
| Bulk update | round(18.41495, 2 dec) = 18.41 | area × 18.41 = 184.10 |

Differenza esatta: `(18.41495 - 18.41) × 10 = 0.0495 ≈ 0.05`.

## Fix

Stesso arrotondamento ovunque:

```ts
unit_cost = Math.round((comprehensive + finish) * 100) / 100;
total_cost = Math.round(area * quantity * unit_cost * 100) / 100;
```

Applicato in:
- `AddToEstimateDialogV2.tsx` (preview live)
- `useCreateEstimateStratigraphy.tsx` (insert)
- `useBulkUpdateEstimateStratigraphyPrices.tsx` (già rounded — riferimento)

## Lezione

**Arrotondamento è parte della "convenzione" del valore stored**. Diversi punti di scrittura con diverso livello di rounding → divergenza inevitabile su comparazioni successive.

Regola d'oro: **scegli precision policy E applicala ovunque**. Per i prezzi UI in €:
- 2 decimali ovunque
- Round PRIMA di moltiplicare per area (non dopo): garantisce che `unit_cost × area = total_cost`.
